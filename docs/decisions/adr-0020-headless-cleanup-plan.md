## Purpose

ADR-0020 mandates a headless (pure-behavior) layer + a swappable skin for every
reusable UX primitive. The adoption audit finds **95 `skinned-without-headless`
components** — skins that own non-trivial behavior with no backing engine — plus
**7 near-duplicate clusters** that should fold onto one engine. This plan
decomposes that backlog into bounded, independent units, each shaped like a
single `/batch` wave unit: one cohesive directory or consolidation cluster,
independently shippable as one branch / one PR.

Each unit, when executed by a future PR, does the same thing: extract the
owned behavior (state, effects, keyboard/event handling, ARIA computation) from
the skin(s) into a colocated pure `use*` engine with a typed contract and pure
colocated tests, then have the skin(s) consume the engine. Consolidation units
additionally fold sibling forks onto the one new engine. None of that happens
here.

## Ranking method

Units are ranked by **leverage** then **risk**:

1. **Cross-area consolidation clusters first.** Folding a fork removes
   divergence *and* creates the first reusable engine for that primitive —
   double payoff, and it directly serves ADR-0010 / ADR-0027.
2. **Highest-reuse / highest-density areas next.** `ui/` and `shared/`
   primitives are imported the most, so an engine there pays back across every
   consumer; large single-area gap counts (project, chat) come next.
3. **Risk descending within a tier.** Self-contained presentational-ish skins
   are lower risk than shells that wire stores, navigation, or live runtime.

**Out of scope for every unit below** (owned by other workstreams or excluded by
ADR-0020): `components/experimental/*.lab.ts`, `components/agent-activity/cockpit/*`
(6 gaps the audit reports but this plan does **not** schedule — concurrent
owners), and `presentation-only` / `support` files (legitimately exempt).

## Sizing legend

- **S** ≈ 1 engine, ≤2 skins, < ~150 LOC moved.
- **M** ≈ 2–4 engines or one consolidation, one directory.
- **L** ≈ 5+ gap files in a directory; may split into 2 PRs along the natural
  sub-grouping noted in the unit.

## Execution notes for future waves

- Regenerate the artifact (`node ActionPlan/quality/audit-headless-adoption.js`)
  at the start of each wave; the `skinned-without-headless` count is the burn-
  down metric. A unit is "done" when its files leave that state.
- Keep engines pure per ADR-0020: no styling/theme imports, no feature/service
  tier imports, no host-runtime. `check-headless-purity.js` will then assert the
  new engines once they are classified headless.
- Each unit is one branch / one PR (one cohesive directory or cluster). Do not
  batch unrelated areas into a single PR.
- When a unit consolidates a cluster, migrate every fork in the same PR (no
  parallel paths), per ADR-0027 single-path and the repo change-management rule.
- Promotion of the audit to a blocking gate is the final follow-up after the
  backlog burns down enough to set a defensible non-zero baseline.
