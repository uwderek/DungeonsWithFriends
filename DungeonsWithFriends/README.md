# Dungeons With Friends - Foundation (React Native Web + Nativewind v4)

This represents the foundational codebase for the Dungeons With Friends project. It is structured using Feature-Sliced Design to build out local-first capabilities with TinyBase-backed persistence.

## 🚀 Getting Started

First, install your dependencies. 

```bash
npm install
```

### Running the App Locally

To start the development server:

```bash
npm start
``` 

Press **`w`** in the terminal to open the web application. You can press `a` or `i` to open an Android or iOS emulator if installed.

To export the web bundle, run:

```bash
npx expo export -p web
```

## Testing Architecture

Run verification directly through the repo scripts and local CLI tools:

```bash
npm run typecheck
npm test -- --runInBand
npm run test:e2e
npm run lint:css
```

For a focused Jest run, pass the test path after `--`:

```bash
npm test -- --runInBand src/shared/store/export-import.test.ts
```

`npm run check` runs TypeScript, Jest, and CSS linting in sequence. Run E2E separately when a change affects browser-level flows.

### CSS Linting

The project uses **Stylelint** with `stylelint-config-standard` for CSS quality enforcement. Configuration lives in `.stylelintrc.json` at the project root. Tailwind directives (`@tailwind`, `@apply`, `@layer`, `@config`) are explicitly whitelisted.

Generated files in `coverage/`, `output/`, and `dist/` are automatically excluded from CSS linting.

### Logging Requirements
To aid debugging during automated tests, application code is expected to log its execution cleanly.
1. **Always prefixed:** `console.log('[SyncProvider] Initializing');`
2. **Captured natively:** Jest and Playwright output should include enough component/runtime context to diagnose the failing assertion.

### Evaluating Test Results
Use the direct command output first. For E2E failures, inspect the Playwright report and traces generated by the local Playwright runner.

### Coverage Expectations
The project maintains a strict minimum baseline of **80% Code Coverage**.
- Use Jest coverage output for line, branch, and function gaps when coverage is relevant to the change.
- New schema, migration, import/export, and non-trivial hook changes should ship with co-located tests.
